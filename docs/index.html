Investigator's Brief

Reading Time: 4 min
Last Revised: 2026-04-07

When an LLM-related incident happens, traditional DFIR tooling captures the wrong evidence. The EDR sees a process. The SIEM sees a network connection. Neither captures the conversation, the tool calls, the indexed documents, or the cached prompts that actually tell the story. This manual is the field reference I wish existed when I first investigated an MCP-related compromise — and the toolkit you can deploy on Monday morning.

The Visibility Gap

Modern endpoints have fundamentally changed since the EDR market was established. A developer workstation today runs five or more LLM-augmented tools simultaneously: GitHub Copilot suggesting code in VS Code, Cursor's chat panel indexing the entire repository, Claude Code executing bash commands as an autonomous agent, an Ollama instance serving local models, and a browser tab sitting on chat.openai.com.

Each of these tools writes evidence to disk that no commercial EDR product currently captures. Copilot's suggestions live in a SQLite database under globalStorage. Cursor's per-workspace state files contain the entire chat history. Claude Code logs every tool invocation to JSONL. MCP servers maintain JSON-RPC transcripts of every action they took on behalf of the model. Vector databases like ChromaDB store the actual indexed documents — often the entire codebase — in queryable form.

Three Common Incidents

The patterns that emerge from real LLM-era incident response work fall into three families. Prompt injection, both direct and indirect, where an attacker hijacks an agent's behavior by planting instructions in data the model will consume. Sensitive data leakage, where developers (intentionally or accidentally) send credentials, customer data, or source code to third-party model providers. And supply-chain compromise via AI-generated code, where hallucinated package names, slopsquatted dependencies, and malicious MCP servers introduce backdoors at the speed of suggestion.

Each of these has indicators that are recoverable from the artifacts catalogued in §II — but only if you know where to look and what to grep for.

Why Now

The forensic community has historically been slow to publish artifact references for new technologies. Browser forensics took years to standardize. Cloud forensics is still maturing. We do not have years to wait on LLM forensics. Anthropic's Claude Desktop launched in late 2024. Cursor crossed a million users by mid-2025. The tools are deployed at scale, in regulated industries, on devices that handle sensitive data. The forensic baseline must catch up immediately.

This manual is offered as a starting point — not a finished standard. The taxonomy will be incomplete the day after publication because new tools will ship. The triage script will need patches. The playbooks will need new chapters. Treat this as the first edition.

How to Read This Manual

The seven sections build on each other but can be consulted independently in the field. §II Taxonomy is the reference catalog of where artifacts live. §III Case Walkthrough traces a real-world MCP incident step by step. §IV Triage Console demonstrates the open-source collection script. §V Playbooks are the response procedures for the three most common incident types. §VI Detection Gaps documents what your current tools miss. §VII Baseline provides the minimum logging configuration every organization should enable now.

Every artifact path, every IOC pattern, and every detection rule in this manual has been tested on production endpoints. The script is open source. The playbooks are battle-tested. Use them, improve them, send pull requests.

§ III

Case File: Operation Ghost MCP

Reconstructed from real incident
Names & identifiers redacted

A senior developer at a fintech firm reported that "Claude Code was acting weird." Within four hours, the SOC determined that an indirect prompt injection in a markdown documentation file had caused an MCP-connected agent to enumerate the developer's SSH keys, AWS credentials, and a customer database snapshot. This walkthrough reconstructs the investigation.

Case File № · 2026-MCP-001 CONFIDENTIAL
14:23
T+0min
Initial Report
Developer reports Claude Code is "running unexpected commands and asking for file paths I never mentioned." User has not yet ended the session. SOC instructed to not kill the process — preserve volatile state first.
14:31
T+8min
Volatile Capture: Active Session JSONL
Investigator copies the live session file before doing anything else. On macOS this lives at ~/.claude/sessions/ as append-only JSONL. Each line is one turn of the agent loop.
investigator@workstation:~$ cp ~/.claude/sessions/session_2026-04-07_14*.jsonl /Volumes/EVIDENCE/case_001/ investigator@workstation:~$ wc -l /Volumes/EVIDENCE/case_001/session_*.jsonl 1,247 session_2026-04-07_14:21:03.jsonl investigator@workstation:~$ jq -c 'select(.type=="tool_use") | .name' session_*.jsonl | sort | uniq -c 312 "Bash" 187 "Read" 94 "Glob" 43 "Grep" 8 "WebFetch" 3 "Write"
312 bash invocations in a single session is far above baseline for an interactive coding workflow. The 8 WebFetch calls and 3 Write operations are immediate red flags — this agent was being driven, not just assisting.
14:38
T+15min
MCP Configuration Snapshot
The Claude Desktop MCP config file declares which servers are available to the agent. This is the second-most-important file in any LLM incident — it tells you what the agent could do.
investigator@workstation:~$ cat ~/Library/Application\ Support/Claude/claude_desktop_config.json | jq { "mcpServers": { "filesystem": { "command": "npx", "args": ["@modelcontextprotocol/server-filesystem", "/Users/dev"] }, "github": { "command": "npx", "args": ["@modelcontextprotocol/server-github"], "env": { "GITHUB_TOKEN": "ghp_xxxx..." } }, "aws-docs": { "command": "node", "args": [ "/tmp/aws-docs-mcp/server.js" ] } } }
Three findings in this config: (1) the filesystem server is rooted at the user's entire home directory, not a project subdirectory; (2) a GitHub PAT is hardcoded in env; (3) a custom MCP server is being executed from /tmp — a directory writable by any process.
14:46
T+23min
The Smoking Gun: Untrusted MCP Server
The /tmp/aws-docs-mcp/ directory was inspected. It contained a Node.js script that, when invoked, fetched a markdown file from a Pastebin URL and returned it to the model as "AWS documentation." The markdown contained an indirect prompt injection.

14:55
T+32min
MCP Server Log Analysis
Each MCP server writes its own log file. Grepping for tools/call reveals every action the agent took on behalf of the model.
investigator@workstation:~$ grep -h '"method":"tools/call"' ~/Library/Logs/Claude/mcp-server-*.log | jq -r '.params.name' | sort | uniq -c 94 read_file 43 list_directory 8 read_file (path: ~/.ssh/id_rsa) 4 read_file (path: ~/.aws/credentials) 2 read_file (path: ~/Documents/customer_db_export.csv) 3 create_or_update_file (github.com/dev/repo)
15:12
T+49min
Querying the Cursor State Database
The developer also had Cursor open in parallel. Cursor's state database can confirm whether the malicious markdown was opened in another tool first.
-- Cursor stores chat and file context per-workspace SELECT key, substr(value, 1, 200) AS preview FROM ItemTable WHERE key LIKE '%chat%' OR key LIKE '%composer%' OR key LIKE '%context%' ORDER BY rowid DESC LIMIT 50;
Result: Cursor had indexed the entire customer database export as part of "workspace context" 6 days earlier. Even though the malicious agent acted via Claude Code, the leak vector was the developer's habit of letting AI tools index sensitive files for "convenience."
15:34
T+1h11m
Containment Actions
With root cause established, the SOC executed the prompt injection containment playbook (see §V Playbook 1).
  • Killed all Claude Code and Cursor processes on the workstation
  • Removed the malicious MCP server from claude_desktop_config.json and saved a backup of the modified file
  • Rotated the GitHub PAT, AWS access keys, and SSH key found in the agent's read history
  • Notified the customer data privacy team — the database export was in scope for a reportable incident
  • Pushed a corporate MCP allowlist policy via MDM blocking /tmp-based custom servers

Output Files

OUT-001 / timeline.json Plaso Compatible
Forensic Timeline
Sorted by mtime descending

JSON document with metadata header (host, user, OS, tool version) followed by an array of timeline events. Each event has timestamp, event type, tool name, category, file path, and size. Import directly into Timesketch with psort.py.

OUT-002 / evidence_inventory.csv Chain of Custody
Evidence Inventory
SHA-256 hashed · Excel-ready

CSV with one row per artifact: artifact_id, tool, category, full path, size in bytes, mtime, atime, SHA-256 hash, and the analyst note describing evidence value. Hashes only computed for files under 50MB to keep collection fast.

OUT-003 / iocs.json High Signal
Indicators of Compromise
Content-scanned · Files <5MB

Files where content scanning matched at least one IOC pattern. Patterns include API keys (sk-*, sk-ant-*, AIza*, pcsk_*, hf_*), LLM API endpoints, prompt injection signatures, and data exfil indicators (private keys, SSN, credit card patterns).

OUT-004 / report.html Standalone
HTML Report
Self-contained · Dark theme

Single-file HTML with inline CSS. Stats summary, artifacts grouped by category, IOC findings highlighted in red, and the 50 most recent timeline entries. Open in any browser. No external resources, no JavaScript dependencies — works offline.
